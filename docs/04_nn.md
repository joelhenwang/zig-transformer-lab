# Chapter 4: Neural Network Modules and Optimizers

This chapter covers the nn (neural network) module system and optimizer
implementations that turn autograd into a trainable system. After reading
this, you'll understand how PyTorch's `nn.Module` and `torch.optim` work
internally.

## Why a Module System?

Stage 3 gave us autograd — we can compute gradients for any tape-recorded
computation. But a raw tape is too low-level for real models. Consider what
a single forward pass through a transformer block requires:

```
LayerNorm(x) → Attention → residual add → LayerNorm → MLP → residual add
```

Each sub-layer has its own weights, and the optimizer needs to find *all*
weights without the user manually listing them. A module system solves two
problems:

1. **Encapsulation**: Each layer owns its parameters and exposes a `forward()`
   method. You compose layers into blocks, blocks into models.
2. **Parameter collection**: Each layer can enumerate its learnable tensors.
   The optimizer iterates over a flat list collected from all layers.

This is exactly what PyTorch's `nn.Module` does — but we implement it without
inheritance or runtime reflection.

## The Module Convention

PyTorch uses `class Linear(nn.Module)` with `self.register_parameter()`. Python's
runtime reflection (`nn.Module.parameters()`) walks `self.__dict__` to find
every `nn.Parameter`.

Zig has no reflection. Instead, we use a **convention** — every layer is a
concrete struct with four methods:

```zig
// Every nn layer must expose these four methods:
pub fn init(allocator, config, rng) !Self       // allocate + initialize weights
pub fn forward(self, input, tape) !Tensor       // compute output, record on tape
pub fn parameters(self, list: *ArrayList(*Tensor)) void  // append learnable tensors
pub fn deinit(self) void                        // free owned tensors
```

No vtable, no inheritance. The model knows its sub-layers by name and calls
`parameters()` on each one.

### Why no vtable for layers?

A vtable adds indirection for every `forward()` call. For layers that are
called thousands of times in a training loop, the overhead matters. More
importantly, each layer has a different `forward()` signature (Embedding
takes integer IDs, Linear takes float tensors), so a common interface would
need `*anyopaque` arguments — losing type safety for no real benefit.

The optimizer *does* use a vtable (see below), because `SGD.step()` and
`AdamW.step()` have identical signatures and the training loop benefits from
swapping optimizers at runtime.

### TransformerConfig

All layers share a configuration struct:

```zig
pub const TransformerConfig = struct {
    vocab_size: usize = 64,      // number of unique tokens
    d_model: usize = 32,        // embedding dimension
    max_seq_len: usize = 16,     // context window length
    d_ff: usize = 128,          // feed-forward hidden size (typically 4 × d_model)
    ln_eps: f32 = 1e-5,         // LayerNorm epsilon
    bias: bool = true,           // whether Linear layers use bias
};
```

Defaults are pedagogically small so the model trains on CPU in seconds.

## Leaf Layers

### Linear — The Workhorse

Every projection in a transformer is a linear layer. It's the single most-
used layer in the architecture.

```
y = x @ W^T + b

Weight: (D_out, D_in)   ← PyTorch convention
Bias:   (D_out,)
```

**Why (D_out, D_in) and not (D_in, D_out)?** PyTorch stores weights as
(D_out, D_in) and computes `x @ W^T`. This convention lets the bias be a
simple (D_out,) vector added to the output. If we stored (D_in, D_out),
we'd compute `x @ W`, which works but means the weight gradient from
`dL/dW = x^T @ dL/dy` would need transposing. The PyTorch convention makes
the gradient shapes align naturally with the parameter shapes.

#### Initialization: Kaiming Uniform

```zig
bound = sqrt(6 / fan_in)   // fan_in = D_in for (D_out, D_in) weight
W ~ U(-bound, bound)
b = 0
```

Kaiming initialization ("He init") preserves activation magnitudes across
layers when using ReLU-family activations. For a weight matrix with fan_in
inputs, drawing from U(-bound, bound) where bound = sqrt(6/fan_in) ensures
the variance of the output is approximately equal to the variance of the
input. This prevents early-layer activations from exploding or vanishing.

The formula comes from "Delving Deep into Rectifiers" (He et al., 2015).
We use the uniform variant because it's the default in PyTorch's
`nn.Linear`.

#### Forward: 3D Support

Linear can accept 2D input `(N, D_in)` or 3D input `(B, T, D_in)`:

```
2D:  input (N, D_in) → output (N, D_out)
3D:  input (B, T, D_in) → flatten to (B*T, D_in) → matmul → reshape to (B, T, D_out)
```

The 3D reshape is tape-tracked via `reshapeTracked()` so the backward can
convert the 2D matmul gradient back to 3D, matching the original input shape.

#### Intermediate tensor lifetime management

The forward pass creates several intermediate tensors:

```
x_flat = reshapeTracked(input, (B*T, D_in))     ← owned copy
wt = transpose2dTracked(weight, (D_in, D_out))  ← owned copy
output_2d = matmul(x_flat, wt)                   ← owned result
bias_2d = reshapeTracked(bias, (1, D_out))       ← owned copy
biased = add(output_2d, bias_2d)                  ← owned result
output_3d = reshapeTracked(biased, (B, T, D_out)) ← owned result
```

Each of these intermediates is `defer`-freed in `forward()`. But the tape's
SavedData stores slices pointing into these buffers. If we free them before
backward, we get use-after-free.

**Solution: `tape.keepAlive()`**. Before each `defer deinit()`, we call
`tape.keepAlive(&intermediate)`, which:

1. Appends the tensor's `data` slice to `tape.kept_alive`
2. Sets `tensor.owned = false` so subsequent `deinit()` is a no-op
3. The tape frees all kept-alive buffers in `tape.deinit()`

This is analogous to PyTorch's "saved tensor" refcounting — PyTorch keeps
intermediate tensors alive as long as the computation graph needs them.
Our mechanism is simpler (no refcounting, just a list) but serves the same
purpose.

```
// Pattern used in every nn forward():
var intermediate = try some_op(alloc, input, tape);
if (tape) |t| try t.keepAlive(&intermediate);  // tape takes ownership
defer intermediate.deinit(alloc);               // no-op because owned=false
```

### Embedding — The Lookup Table

An embedding layer maps discrete token IDs to continuous vectors. It's the
simplest possible "layer" — just a lookup table.

```
Forward:  output[i] = weight[idx[i]]           (gather rows)
Backward: grad_weight[idx[i]] += grad_output[i] (scatter-add)
```

Shape:
```
Weight: (vocab_size, d_model)
Input:  (N,) integer IDs  → Output: (N, d_model)
Input:  (B, T) integer IDs → Output: (B, T, d_model)
```

#### Why not a fused OpKind?

The embedding forward is just a memory copy — no arithmetic. But the
backward is special: it's a scatter-add (multiple positions can reference
the same vocabulary row, so their gradients must accumulate). This doesn't
fit any existing op's backward pattern, so we added a dedicated `.embedding`
OpKind in Stage 3.

The backward implementation in `backward.zig`:

```zig
fn backwardEmbedding(node, grad_output, tape) {
    const info = node.saved.embedding_info;
    // For each position i:
    //   grad_weight[indices[i]] += grad_output[i]
    for (0..n_ids) |i| {
        const idx = @intFromFloat(info.indices[i]);
        const src_offset = i * d_model;
        const dst_offset = idx * d_model;
        for (0..d_model) |j| {
            grad_weight.data[dst_offset + j] += grad_output.data[src_offset + j];
        }
    }
}
```

Key detail: parent 0 is the weight (gets gradient), parent 1 is null (the
indices are integer — no gradient needed).

#### Initialization

```zig
std_dev = 1 / sqrt(d_model)
W ~ N(0, std_dev)
```

The standard deviation scales with d_model so that larger embedding
dimensions don't produce larger activations. This matches the default
in PyTorch's `nn.Embedding`.

### LayerNorm — Normalization as Composable Ops

Layer normalization stabilizes training by normalizing each position's
activation to zero mean and unit variance, then applying a learnable affine
transform.

```
mu = mean(x, axis=-1)                    // (..., 1)
x_centered = x - mu                      // (..., D)
var = mean(x_centered², axis=-1)         // (..., 1)
inv_std = 1 / sqrt(var + eps)            // (..., 1)
x_norm = x_centered * inv_std            // (..., D)
y = gamma * x_norm + beta               // (..., D)
```

Parameters:
```
gamma: (D,) initialized to 1.0   (scale)
beta:  (D,) initialized to 0.0   (shift)
```

When gamma=1 and beta=0, LayerNorm is the identity transform. The network
learns to deviate from identity as needed. This initialization ensures
LayerNorm doesn't disrupt the signal flow early in training.

#### Why compose from ops instead of a fused OpKind?

We *could* add a `.layernorm` OpKind with a custom backward. But composing
from existing ops is more pedagogical:

1. Students see *exactly* how each gradient flows through ~7 tape nodes
2. The same pattern (compose ops instead of fusing) applies to any new
   layer you build
3. No need to derive and implement a custom backward — autograd handles it

The tradeoff is performance: a fused LayerNorm kernel is ~10x faster because
it does one pass over the data instead of seven. But this is a pedagogical
project, and CPU training on tiny models doesn't need the speed.

For the CUDA stage, we *will* implement a fused LayerNorm kernel
(`layernorm.cu`) because the 7-pass approach is too slow on GPU.

#### Broadcasting subtlety

The mean reduces the last axis, producing shape `(..., 1)`. When we compute
`x - mu`, the subtraction broadcasts mu along the last dimension. The tape's
`backwardSub` handles this via `sumToShape()` — it sums the gradient along
the broadcast axis to match the original shape.

Similarly, gamma is `(D,)` but we reshape it to `(1, D)` via
`reshapeTracked()` before multiplying. The backward for `mul` then calls
`sumToShape()` to reduce the gradient back to `(D,)` for gamma.

### GELU — Activation Wrapper

The Gaussian Error Linear Unit is the activation function used in GPT-2:

```
GELU(x) = x * Φ(x) = x * 0.5 * (1 + erf(x / sqrt(2)))
```

Where Φ is the CDF of the standard normal distribution. GELU is similar to
ReLU but smoother — it's non-monotonic for negative values, which helps the
network learn more expressive representations.

Our GELU is a thin wrapper over `ops.unary.geluExact()`, which computes the
exact formula using an erf approximation. The wrapper is stateless — it
stores only the allocator (for compatibility with the module convention).

## Composite Layers

### MLP — Feed-Forward Network

```
MLP(x) = fc2(GELU(fc1(x)))

fc1: (D, d_ff)   — expand from D to 4D
GELU: nonlinearity
fc2: (d_ff, D)   — project back from 4D to D
```

The MLP is where most of the transformer's parameters live. With d_ff = 4×d_model,
the two weight matrices have 2 × D × 4D = 8D² parameters, compared to the
attention's 4D² (for Q, K, V, O projections).

This 2:1 ratio (MLP:Attention params) is a deliberate design choice from the
original transformer paper. The attention learns *where* to look (routing),
while the MLP learns *what* to compute (transformation).

### CausalSelfAttention — The Heart of the Transformer

Single-head causal self-attention implements the core mechanism:

```
Q = X @ W_q^T    (B, T, D) — queries
K = X @ W_k^T    (B, T, D) — keys
V = X @ W_v^T    (B, T, D) — values

scores = Q @ K^T / sqrt(D)  (B, T, T) — attention scores
scores += causal_mask        (B, T, T) — mask future positions
weights = softmax(scores)    (B, T, T) — attention weights
attn_out = weights @ V       (B, T, D) — weighted sum of values

output = attn_out @ W_o^T   (B, T, D) — output projection
```

#### QKV Projections

Four Linear layers: W_q, W_k, W_v (input projections) and W_o (output
projection). Each maps D → D (single head). In multi-head attention
(Stage 8), these would map D → D with internal reshaping.

#### Causal Mask

The causal mask prevents position i from attending to position j > i:

```
mask[i][j] = 0     if j <= i   (allowed: present and past)
mask[i][j] = -1e9  if j > i    (forbidden: future)
```

After adding this mask, softmax assigns ~0 probability to future positions
(because e^{-1e9} ≈ 0). The mask is precomputed for `max_seq_len`
positions in `init()` and sliced to the actual sequence length T in
`forward()`.

Why -1e9 and not -inf? Numerically, e^{-inf} = 0 exactly, but adding
-inf to a finite score can produce NaN if the score is also very large.
-1e9 is large enough to zero out the attention weight while remaining
finite. This is the standard practice in GPT-2 and nanoGPT.

#### Scaled Dot-Product

The 1/sqrt(D) scaling prevents the dot products from growing too large
as D increases. Without scaling, Q·K values of magnitude ~D would push
softmax into saturation (near-one-hot), making gradients vanish.

This is from the original attention paper: "We suspect that for large
values of d_k, the dot products grow large in magnitude, pushing the
softmax function into regions where it has extremely small gradients."

#### Batched Matmul

Q, K, V are 3D tensors (B, T, D). The attention computation uses
`matmulBatch()` which treats the first dimension as the batch index
and performs B independent 2D matmuls.

For K^T, we use `transposeInner2d()` which swaps the last two dimensions
of a 3D tensor: (B, T, D) → (B, D, T). This is tracked on the tape so
backward can transpose the gradient back.

### TransformerBlock — Pre-Norm Residual

```
h = x + Attention(LN_1(x))
out = h + MLP(LN_2(h))
```

The pre-norm architecture (LayerNorm *before* attention/MLP) is more
stable for training than post-norm (LayerNorm *after*). In post-norm,
the residual path goes through LayerNorm, which can squash the gradient
signal. In pre-norm, the residual is a clean identity + transformation,
so gradients flow unimpeded through the skip connection.

```
Pre-norm:  x → LN → Attn → (+x) → LN → MLP → (+h) → output
Post-norm: x → Attn → (+x) → LN → MLP → (+h) → LN → output
```

GPT-2 uses pre-norm, and we follow that convention (decision D13).

#### Residual Connections

The `add(input, attn_output)` records a tape node. During backward:

```
dL/dinput = dL/dadd_output * 1     (from the identity path)
dL/d_attn = dL/dadd_output * 1     (from the transformation path)
```

Both gradients are identical to the upstream gradient. This is the
"gradient highway" that makes deep networks trainable — the gradient
from the loss can reach early layers without being multiplied by
many small factors.

## The Full Model: TinyWordTransformer

```
tok_embed(ids) + pos_embed(positions)   → (B, T, D)
block(x)                                → (B, T, D)
ln_f(x)                                → (B, T, D)
lm_head(x)                             → (B, T, V)  logits
```

### Token + Positional Embeddings

Token embeddings convert integer IDs to vectors. Positional embeddings
add position information — without them, the model is permutation-
invariant (it can't distinguish "cat sat" from "sat cat").

```
x = tok_embed(ids) + pos_embed([0, 1, ..., T-1])
```

The positional IDs are [0, 1, ..., T-1], created fresh each forward
pass. Position embeddings are learned (not sinusoidal), following GPT-2.

### lm_head — Output Projection

The final Linear layer maps from D to V (vocabulary size) without bias.
This produces the logits used by cross-entropy loss. No bias is used
because the token embeddings already encode a per-token offset.

### Parameter Count

For V=32, D=16, T=8, d_ff=64, with bias:

```
tok_embed.weight:    32 × 16 = 512
pos_embed.weight:     8 × 16 = 128
ln1.gamma + beta:         32
attn: 4 × (16×16 + 16) = 1088   (4 Linear layers with bias)
ln2.gamma + beta:         32
mlp.fc1: 64×16 + 64 = 1088
mlp.fc2: 16×64 + 16 = 1040
ln_f.gamma + beta:        32
lm_head.weight:    16 × 32 = 512
                               --------
                       Total: ~4,464 parameters
```

A tiny model — but enough to verify that the training pipeline works.

### Save/Load — Checkpoint Format

The model can save and load binary checkpoints:

```
[TWTL]       — 4-byte magic ("TinyWordTransformer Lab")
[u32: 1]     — version number
[u32: N]     — number of parameters
For each parameter:
  [u32] name_length
  [bytes] name (UTF-8)
  [u8]   ndim
  [4×u32] dimensions (padded with 0)
  [u32]  data_length (in bytes)
  [f32×] data (little-endian)
```

Parameters are matched by name, so the checkpoint is resilient to
reordering. Unknown parameters are skipped (forward-compatible).

## Optimizers

### The Optimizer Vtable

The optimizer interface uses a vtable pattern for runtime polymorphism:

```zig
pub const Optimizer = struct {
    ctx: *anyopaque,              // pointer to concrete optimizer (SGD, AdamW)
    step_fn: StepFn,              // function pointer: update parameters
    zero_grad_fn: ZeroGradFn,     // function pointer: zero gradients
    deinit_fn: DeinitFn,          // function pointer: free state

    pub fn step(self, params) !void { ... }
    pub fn zeroGrad(self, params) void { ... }
    pub fn deinit(self, allocator) void { ... }
};
```

Usage:

```zig
var adam = try AdamW.init(alloc, .{ .lr = 3e-4 });
var opt = adam.optimizer();         // wraps AdamW in Optimizer vtable
try opt.step(params.items);         // calls AdamW.step internally
opt.zeroGrad(params.items);
opt.deinit(alloc);                  // frees AdamW state + struct
```

### SGD with Momentum

```
v = momentum * v + grad + weight_decay * param
param -= lr * v
```

**Coupled weight decay**: the weight decay term is added to the gradient
before momentum. This is the classical SGD convention. It means the weight
decay effect depends on the learning rate — if you increase lr, you also
increase the regularization strength.

Config defaults:

```zig
SGDConfig {
    lr: f32 = 0.01,
    momentum: f32 = 0.0,     // 0 = vanilla SGD, 0.9 = standard
    weight_decay: f32 = 0.0,
}
```

#### Velocity buffers

The momentum buffer `v` is a per-parameter tensor stored in a HashMap
keyed by the parameter's data pointer. On the first step, v is initialized
to zeros. On subsequent steps, v is updated in-place.

```zig
velocity: AutoHashMap(usize, Tensor)   // key = @intFromPtr(param.data.ptr)
```

The key is the pointer address, not the parameter index. This works because
each parameter has a unique heap allocation. If two parameters shared the
same data buffer (e.g., weight tying), they'd collide — but our model
doesn't use weight tying.

### AdamW — The Transformer Optimizer

```
m = β₁ * m + (1 - β₁) * grad              (1st moment — exponential moving avg of gradients)
v = β₂ * v + (1 - β₂) * grad²              (2nd moment — exponential moving avg of squared gradients)
m̂ = m / (1 - β₁ᵗ)                          (bias-corrected 1st moment)
v̂ = v / (1 - β₂ᵗ)                          (bias-corrected 2nd moment)
param -= lr * (m̂ / (√v̂ + ε) + wd * param)  (AdamW update)
```

#### Why AdamW and not Adam?

Standard Adam adds weight decay to the gradient (coupled), which interacts
with the adaptive learning rate. AdamW applies weight decay *directly* to
the parameter (decoupled), which:

1. Decouples the weight decay strength from the learning rate
2. Produces better generalization (the original paper shows this)
3. Is the standard for transformer training (GPT-2, GPT-3, LLaMA)

The difference is subtle but important:

```
Adam:  grad_effective = grad + wd * param    (decay added to gradient)
       param -= lr * m̂ / (√v̂ + ε)           (adaptive update on effective grad)

AdamW: param -= lr * (m̂ / (√v̂ + ε) + wd * param)  (decay applied directly)
```

In Adam, the weight decay is scaled by the adaptive learning rate
(m̂ / √v̂), which means the decay strength varies per-parameter. In AdamW,
the weight decay is a constant `lr * wd` applied uniformly.

#### Bias Correction

The moments m and v are initialized at zero, so in early steps they're
biased towards zero. The bias correction compensates:

```
m̂ = m / (1 - β₁ᵗ)
v̂ = v / (1 - β₂ᵗ)
```

At step t=1 with β₁=0.9: `1 - β₁¹ = 0.1`, so `m̂ = m / 0.1 = 10 * m`.
This amplifies the first gradient, compensating for m starting at zero.
As t grows, `β₁ᵗ → 0` and `1 / (1 - β₁ᵗ) → 1`, so the correction fades.

We compute β₁ᵗ by repeated multiplication (a loop of t iterations) because
`std.math.pow` doesn't exist in Zig 0.16.0. For typical training runs
(t < 100,000), this is negligible cost.

#### Per-parameter state

```zig
state: AutoHashMap(usize, ParamState)   // keyed by data pointer

ParamState = struct {
    m: Tensor,   // 1st moment (same shape as param)
    v: Tensor,   // 2nd moment (same shape as param)
}
```

Each parameter gets its own m and v tensors, initialized to zeros on first
use. These persist across steps (unlike the tape, which is recreated each
step).

Config defaults:

```zig
AdamWConfig {
    lr: f32 = 1e-3,
    beta1: f32 = 0.9,         // 1st moment decay
    beta2: f32 = 0.95,        // 2nd moment decay (0.95 not 0.999 — pedagogically faster)
    eps: f32 = 1e-8,          // numerical stability
    weight_decay: f32 = 0.1,  // decoupled weight decay
}
```

We use β₂=0.95 instead of the standard 0.999 because our training runs
are short (50-500 steps). With β₂=0.999, the 2nd moment adapts too slowly
for the optimizer to be effective in so few steps.

## The Training Loop

Example 04 demonstrates the complete pipeline:

```zig
for (0..num_steps) |step| {
    // 1. Fresh tape for this step
    var tape = Tape.init(allocator);
    defer tape.deinit();

    // 2. Track all parameters as tape leaves
    for (params.items) |param| {
        param.requires_grad = true;
        _ = try tape.trackLeaf(param);
    }

    // 3. Forward pass — records ops on tape
    var logits = try model.forward(ids, &tape);
    try tape.keepAlive(&logits);
    // ... reshape, compute loss ...

    // 4. Backward pass — computes dL/dparam for all params
    try tape.backward(&loss);

    // 5. Optimizer step — updates param.data using param.grad
    try opt.step(params.items);
    opt.zeroGrad(params.items);
}
```

### Why a fresh tape each step?

The tape records the forward computation. After backward, all gradients are
computed and stored in `param.grad`. The tape's job is done. Creating a new
tape each step:

1. Frees all tape nodes and kept-alive buffers automatically
2. Avoids stale node IDs from the previous step
3. Matches PyTorch's behavior (dynamic graph rebuilt each forward)

**Critical pitfall**: `trackLeaf()` must always create a fresh node, even
if `param.tape_node` already has a value from a previous step. Old tape_node
IDs from destroyed tapes are invalid — reusing them causes ID collisions
that corrupt the gradient map.

### Loss shape

Cross-entropy expects logits of shape (N, V) and targets of shape (N,).
The model outputs (B, T, V), so we reshape to (B*T, V) before computing
loss:

```zig
const logits = logits_3d.reshape(Shape.init2D(B * T, V));  // view, not copy
```

This reshape is NOT tape-tracked — it's a view. The gradient flows through
the 3D logits which are already on the tape from the model's forward pass.

## KeepAlive: Preventing Use-After-Free

The most subtle bug in the entire nn system is the lifetime of intermediate
tensors during backward. Consider this simplified scenario:

```
forward():
    var a = try add(alloc, x, y, tape);    // a.data = heap_alloc(256)
    try tape.keepAlive(&a);                 // tape takes ownership of a.data
    defer a.deinit(alloc);                   // no-op because a.owned = false
    return a;                                // caller gets a with owned=false

backward():
    // Tape's SavedData has .tensor_pair = {x, y} — snapshots of the inputs.
    // The add node doesn't save `a`, it saves the inputs.
    // But `x` and `y` might also be intermediates from other ops...
    // Their data slices point into buffers that must still be alive.
```

The rule is: **any tensor whose data is referenced by a tape node's SavedData
must outlive the backward call**. The `keepAlive` mechanism ensures this by
transferring buffer ownership from the tensor to the tape.

Without keepAlive, the `defer deinit()` in forward() would free the buffer
before backward runs. The SavedData slice would point to freed memory.
This is the Zig equivalent of Python's reference counting keeping tensors
alive as long as the computation graph holds a reference to them.

### When is keepAlive needed?

Every intermediate tensor in a layer's `forward()` that is:
1. Created by an op (not a parameter — parameters outlive the model)
2. Referenced by a downstream op's SavedData
3. `defer`-freed before backward runs

In practice, this means **every** intermediate in every nn forward method.
It's verbose but safe.

| `opt.zeroGrad(params)` | `optimizer.zero_grad()` |
| `tape.keepAlive(&tensor)` | (automatic in PyTorch via refcounting) |

## Common Mistakes

### 1. Forgetting keepAlive on intermediates

**Symptom**: Segfault or garbage gradients during backward.

**Cause**: An intermediate tensor is freed before backward runs. The tape's
SavedData contains a `data` slice pointing to freed memory.

**Fix**: Add `if (tape) |t| try t.keepAlive(&intermediate)` before every
`defer intermediate.deinit()` in forward methods.

### 2. Using trackLeaf with stale tape_node

**Symptom**: Gradient corruption — two different tensors get the same
gradient, or gradients accumulate incorrectly.

**Cause**: After step N, `param.tape_node` still holds the node ID from
the destroyed tape. Step N+1 creates a new tape, and `trackLeaf()` short-
circuits because `tape_node` is non-null, returning a stale ID that might
collide with another node.

**Fix**: `trackLeaf()` must always create a fresh node, ignoring the
existing `tape_node` value. The ID from the previous tape is invalid.

### 3. Reshape vs reshapeTracked

**Symptom**: Shape mismatch during gradient accumulation.

**Cause**: `Tensor.reshape()` creates an untracked view. The backward for
a downstream op produces a gradient that doesn't match the original
parameter shape because the reshape wasn't recorded.

**Fix**: Use `reshapeTracked()` when the reshape needs to be reversed
during backward (e.g., 3D→2D in Linear, 1D→2D for broadcasting).

### 4. Forgetting to zero gradients between steps

**Symptom**: Gradients grow without bound; loss oscillates or diverges.

**Cause**: Gradients from step N are still in `param.grad` when step N+1
computes its gradients. The `accumulateGrad()` function adds to existing
gradients (+=), so they double each step.

**Fix**: Call `opt.zeroGrad(params)` after each optimizer step.

### 5. Using transpose2dTracked for temporary tensors

**Symptom**: Use-after-free — the tape saves a snapshot of the transposed
copy, but the copy is freed after forward() returns.

**Cause**: `transpose2dTracked()` creates an owned copy. If you `defer
deinit()` it, the buffer is freed before backward. The tape's SavedData
points to freed memory.

**Fix**: Use `Tensor.transpose2d()` (view) when the original buffer
outlives the tape (e.g., weight tensors owned by the model). Use
`transpose2dTracked()` + `keepAlive()` for intermediates that are freed
in forward().

### 6. Not setting requires_grad on parameters

**Symptom**: All gradients are null; optimizer has nothing to update.

**Cause**: `tape.trackLeaf()` creates a node, but the op only records if
the input has `requires_grad = true`. If parameters don't have this flag,
no backward path is created.

**Fix**: Set `param.requires_grad = true` before `tape.trackLeaf(param)` in
the training loop.

### 7. Reading loss after deinit

**Symptom**: Garbage loss value or segfault.

**Cause**: `loss.deinit()` frees the loss tensor's data buffer, then you
try to read `loss.data[0]`.

**Fix**: Save the loss value before deinit: `const loss_val = loss.data[0]`
then `loss.deinit()`.

### 8. Mismatched optimizer params and model params

**Symptom**: Some parameters aren't updated; training is slow or stuck.

**Cause**: The params list passed to the optimizer doesn't include all
model parameters (e.g., forgot to call `ln_f.parameters()`).

**Fix**: Use `model.parameters(&list)` which collects from all sub-layers.

### 9. AdamW beta2 too high for short runs

**Symptom**: Loss doesn't decrease in the first 50-100 steps.

**Cause**: With β₂=0.999, the 2nd moment v adapts very slowly. It takes
~1000 steps for the bias correction to become negligible. In our short
training runs (50-500 steps), the optimizer is still "warming up".

**Fix**: Use β₂=0.95 (our default) for pedagogically short runs. Switch
to 0.999 for longer training (Stage 6+).

### 10. Embedding backward with duplicate indices

**Symptom**: Incorrect gradients for vocabulary rows that appear multiple
times in the input.

**Cause**: The scatter-add in `backwardEmbedding` correctly accumulates
gradients for duplicate indices. But if the grad_weight tensor is not
zeroed between backward calls, old gradients persist.

**Fix**: The tape's `zeroGrad()` zeros all leaf gradients before backward.
Make sure this is called (it's called automatically in `tape.backward()`
for leaves that are tracked).
